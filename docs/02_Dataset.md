## 2. How Synthea Generates Data

Synthea simulates patients using a **state machine** — a model where each patient moves
through a series of health states over time. For example, a patient might progress through:

```
Healthy → Pre-diabetic → Type 2 Diabetic → Diabetic with Complications → Death
```

Each transition between states has a probability, and each state triggers specific
clinical events:

- Entering the "Annual Physical" state generates an Encounter resource
- The "Vital Signs" module inside that state generates Observation resources
  for blood pressure, body weight, and heart rate
- The "Diabetes Diagnosis" state generates a Condition resource and a
  MedicationRequest for Metformin

Synthea's modules are written in JSON and cover dozens of conditions including:
- Cardiovascular disease (hypertension, coronary artery disease, heart failure)
- Metabolic disorders (diabetes type 2, obesity, hyperlipidaemia)
- Respiratory conditions (asthma, COPD, pneumonia)
- Mental health (depression, anxiety)
- Cancer (breast, prostate, colorectal)
- Musculoskeletal (osteoporosis, back pain)

The simulation runs from birth to death or to the current date, whichever comes first.
A patient born in 1920 who is still alive would have over 100 years of simulated history.

## 4. Table: encounters_ready.csv

**Rows:** 57,667  
**Columns:** 12  
**Model:** PAR (Probabilistic AutoRegressive)  
**Relationship:** Each row references one patient (N:1 to patients)

An encounter is a healthcare visit. One patient can have many encounters — hence the
"many-to-one" relationship (many encounters : one patient).

### Column Reference

| Column | Type | Example | Description |
|---|---|---|---|
| `encounter_id` | UUID | `f762873a-313c-e755-9f13-46efb1b2d7f4` | Unique encounter identifier |
| `patient_id` | UUID | `0005a479-d128-178d-4aac-fefe1b0cc77e` | Foreign key → patients.patient_id |
| `class_code` | categorical | `AMB` | Encounter class from FHIR ActEncounterCode: AMB=ambulatory, EMER=emergency, IMP=inpatient, VR=virtual |
| `type_display` | categorical | `Well child visit (procedure)` | Human-readable description of the encounter type |
| `organization_display` | categorical | `other` | Healthcare organisation where the encounter occurred. Capped to top-50 + "other" to control cardinality |
| `location_display` | categorical | `other` | Specific location (clinic, ward) within the organisation |
| `reason_display` | categorical or NaN | `NaN` | Clinical reason for the visit (may be null if no specific reason coded) |
| `discharge_disposition` | categorical or NaN | `NaN` | How the patient left: discharged home, transferred, etc. Only applies to inpatient encounters |
| `days_since_birth` | float | `0.0` | Temporal position of the encounter: (encounter_start_date − patient_birth_date) in days. First encounter in life = 0.0 |
| `encounter_duration_hours` | float | `0.25` | Length of the encounter in hours. 0.25 = 15 minutes (typical short outpatient visit) |
| `sequence_index` | integer | `0` | 0-based position of this encounter in the patient's chronological encounter history. First encounter = 0, second = 1, etc. |
| `days_since_prev_encounter` | float or NaN | `NaN` | Days since the patient's previous encounter. NaN for the first encounter |

### Why Encounters Use PAR (Not CTGAN)

Encounters are **sequential** events in a patient's life. The time between encounter #1
and encounter #2 is not random — it depends on the patient's health status. A patient
with chronic disease has more frequent visits than a healthy patient. A patient in their
80s has more visits than a patient in their 20s.

Standard generative models like CTGAN treat each row as an independent sample. They
cannot model the relationship between row N and row N+1 for the same patient. PAR
(Probabilistic AutoRegressive) is specifically designed for sequential data: it learns
the conditional distribution of visit N+1 given visits 1 through N.

See [04_Model_Architecture.md](04_Model_Architecture.md) for a full explanation.

## 5. Table: observations_ready.csv

**Rows:** 303,696  
**Columns:** 9  
**Model:** CTGAN  
**Relationship:** Each row references one patient and one encounter

Observations are the measurements taken during a visit — vital signs, lab results,
clinical assessments. This is by far the largest table because a single encounter
can generate 5–20 individual observations.

### Column Reference

| Column | Type | Example | Description |
|---|---|---|---|
| `observation_id` | UUID | `7cb1cdd2-da2c-c523-bda5-b0ee881465ce` | Unique observation identifier. Blood pressure components have a suffix (`_c0`, `_c1`) because they were split from a single BP panel into two separate rows |
| `patient_id` | UUID | `0005a479-d128-178d-4aac-fefe1b0cc77e` | Foreign key → patients.patient_id |
| `encounter_id` | UUID | `f762873a-313c-e755-9f13-46efb1b2d7f4` | Foreign key → encounters.encounter_id |
| `category` | categorical | `vital signs` | Observation category from FHIR. Common values: `vital signs`, `laboratory`, `survey` |
| `loinc_display` | categorical | `Body Height` | Human-readable name for the type of measurement, derived from the LOINC code. This is the key feature that identifies what was measured |
| `value_quantity` | float | `99.1` | The numerical measurement value. Interpreted in the context of `loinc_display` and `value_unit` |
| `value_unit` | categorical | `cm` | Unit of measurement: `cm`, `kg`, `mm[Hg]`, `%`, `g/dL`, etc. |
| `days_since_birth` | float | `0.0` | When the observation was taken: (observation_date − patient_birth_date) in days |
| `sequence_index` | integer | `0` | Position of this observation in the patient's overall observation history |

### The LOINC Filter

The raw observations table contains measurements coded with hundreds of different LOINC
codes. Many of these codes appear only a handful of times in 1,000 patients — for example,
a rare genetic test ordered for one patient.

A machine learning model cannot reliably learn the distribution of a measurement that
appears only 5 times in the training data. To prevent this, Phase 3 preprocessing applies
a frequency filter:

- Keep only LOINC codes that appear at least **100 times** across all patients
- From those, keep only the **top 30 by frequency**

This reduces the observation space from hundreds of LOINC types to 30 well-represented
measurement types, while retaining 303,696 records — which represents the 30 most
clinically significant measurements in routine care.

### Blood Pressure Expansion

Blood pressure is a special case. In FHIR, a blood pressure reading is stored as a
single Observation with a "component" structure containing both systolic and diastolic
values:

```json
{
  "resourceType": "Observation",
  "code": {"coding": [{"display": "Blood Pressure Panel"}]},
  "component": [
    {
      "code": {"coding": [{"code": "8480-6", "display": "Systolic Blood Pressure"}]},
      "valueQuantity": {"value": 128, "unit": "mm[Hg]"}
    },
    {
      "code": {"coding": [{"code": "8462-4", "display": "Diastolic Blood Pressure"}]},
      "valueQuantity": {"value": 82, "unit": "mm[Hg]"}
    }
  ]
}
```

For synthesis, we need each measurement to be a separate row with its own LOINC code and
numerical value. Phase 3 expands these component observations, creating:

- One row for `Systolic Blood Pressure` with `value_quantity = 128`
- One row for `Diastolic Blood Pressure` with `value_quantity = 82`

The original single-row panel is discarded, and the two component rows receive IDs
suffixed with `_c0` and `_c1` to keep them unique.

## 8. Table Relationships and Cardinality

**Cardinality** describes the numerical relationship between tables. For example, "one
patient can have many encounters" is a "one-to-many" relationship.

```
patients ──────────────────────────────────────────────────────┐
  1 patient                                                     │
     │                                                          │
     │  1:many                                                  │
     ▼                                                          │
encounters (57.8 per patient on average)                        │
  1 encounter                                                    │
     │                                                          │
     │  1:many              conditions    medications           │
     ▼                      (1:many)      (1:many)             │
observations               (per patient)  (per patient)         │
(303,696 / 57,667 =                                            │
 ~5.3 per encounter)      All reference patients ──────────────┘
```

### Average Records per Patient

| Table | Total Rows | Per Patient (avg) |
|---|---|---|
| encounters | 57,667 | 57.8 |
| observations | 303,696 | 304.3 |
| conditions | 37,835 | 37.9 |
| medications | 46,734 | 46.8 |

These averages hide significant variation. Some patients (elderly with multiple chronic
conditions) have thousands of records; some (young and healthy) have dozens.

## 9. Why Observations Are the Largest Table

Observations are 6.8× larger than encounters because every single encounter generates
multiple observations. A routine annual physical examination in Synthea triggers:

| Observation | LOINC Code |
|---|---|
| Body Height | 8302-2 |
| Pain severity (score) | 72514-3 |
| Body Weight | 29463-7 |
| Body Mass Index (BMI) | 39156-5 |
| Diastolic Blood Pressure | 8462-4 |
| Systolic Blood Pressure | 8480-6 |
| Heart rate | 8867-4 |
| Respiratory rate | 9279-1 |

That is 8 observations for a single 15-minute visit. A patient with 60 annual physicals
over their lifetime generates 480 observations from physicals alone — before counting
laboratory tests, specialist visits, and emergency encounters.

This scale (303,696 rows) was the direct cause of the Kaggle SIGKILL crash in Phase 4A.
See [09_Engineering_Log.md](09_Engineering_Log.md) for the full incident report.

## 10. Why Encounters Use Sequential Modelling (PAR)

Consider two patients:

**Patient A:** Healthy 25-year-old. One annual physical per year. Encounters at days 9,125
and 9,490 from birth. Gap: 365 days.

**Patient B:** 75-year-old with heart failure, diabetes, and hypertension. Encounters
every 2–4 weeks. Encounters at days 27,375, 27,389, 27,403, 27,420 from birth.
Gaps: 14, 14, 17 days.

A naive generative model (like CTGAN) would treat each encounter row independently.
It would not know that the next encounter for Patient B should come soon, or that the
next encounter for Patient A is a year away. It might generate a visit for Patient B
that happens 400 days after the previous one — clinically unrealistic for a patient
with active heart failure.

PAR (Probabilistic AutoRegressive) learns this temporal dependency. It sees the sequence
of encounters for each patient and learns: "given that this patient had an encounter today
and the previous gap was 14 days, the next gap is likely to be 14–21 days."

### Why Observations Do NOT Use PAR

With 303,696 rows across 998 patients, each patient has ~304 observations on average.
PAR must process entire patient sequences to train. For 998 patients with 304 observations
each, PAR would need to process 304-element sequences — far longer than the typical
sequential model target. This is computationally prohibitive on available hardware.

CTGAN is appropriate for observations because the key distributional question is: "what
is the distribution of blood pressure values for a 50-year-old with hypertension?" This
is a conditional distribution that does not depend on what the previous observation was.

## 11. Column-Level Coding Systems

Three medical coding systems appear in the dataset:

### LOINC (Logical Observation Identifiers Names and Codes)

Used for: `observations.loinc_display`

LOINC is maintained by the Regenstrief Institute. It provides a universal vocabulary for
laboratory tests, clinical measurements, and survey instruments. Each code uniquely
identifies what was measured and in what context.

Examples from this dataset:
- `Body Height` = LOINC 8302-2
- `Systolic Blood Pressure` = LOINC 8480-6
- `Hemoglobin A1c/Hemoglobin.total in Blood` = LOINC 4548-4 (key diabetes monitoring test)

### SNOMED CT (Systematized Nomenclature of Medicine — Clinical Terms)

Used for: `conditions.snomed_display`

SNOMED CT is the world's most comprehensive clinical terminology, maintained by SNOMED
International. It contains over 350,000 active concepts for diseases, procedures, findings,
and situations.

Examples from this dataset:
- `Diabetes mellitus type 2` = SNOMED 44054006
- `Essential hypertension` = SNOMED 59621000
- `Medication review due (situation)` = SNOMED 314529007

### RxNorm

Used for: `medications.rxnorm_display`

RxNorm is maintained by the US National Library of Medicine. It provides normalised names
for US clinical drugs — including dose form, strength, and route.

Examples from this dataset:
- `Metformin 500 MG Oral Tablet`
- `Acetaminophen 325 MG Oral Tablet`
- `Atorvastatin 80 MG Oral Tablet`

### Why We Use Display Names, Not Codes

In the `*_ready.csv` files, we store the human-readable display text rather than the
numeric code (e.g., `"Body Height"` not `"8302-2"`). This is intentional:

1. **Synthesis value:** A generative model can learn that `"Body Height"` values should
   cluster around 160–185 cm for adults. The code `"8302-2"` conveys no numerical meaning.
2. **Interpretability:** A researcher reading the synthetic data understands `"Body Height"`
   immediately; they would need to look up `"8302-2"`.
3. **Redundancy elimination:** Keeping both `loinc_code` and `loinc_display` would double
   the information for no additional modelling benefit.
4. **Cardinality:** The display text is the natural category — grouping by `loinc_display`
   produces the same groups as grouping by `loinc_code`.
